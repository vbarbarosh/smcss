# Full project audit — smcss v1.5.1

> **Status: closed 2026-07-14.** Every finding is fixed, resolved as
> intentional (with a *Resolution* note inline), or shelved with
> measurements (Phase 6). Kept as the record of the changes that followed
> and for the Phase 6 performance analysis.

Date: 2026-07-13. Baseline: commit a2a495b, branch master.
Environment: node v24.14.1, dart-sass 1.72.0 (from lockfile), PHP **not installed**.
Method: every issue below was reproduced in this environment (commands shown);
claims from the earlier `notes/01..07` analysis were re-verified, not copied.

## What is healthy (verified)

- `npm test` → **468 passing** (one golden-table suite; see caveats in A2/A3).
- `dist/sm.css` is **byte-identical** to a fresh build (`sass demos/sm.sass | postcss`
  with `.browserslistrc` honored), as are `dist/reset.css` and `demos/demo.css`.
  No src↔dist drift today.
- `npm pack --dry-run`: 57 files, 72.9 kB packed; `files` field works as intended —
  no test/demo/doc leakage. `main: index.sass` resolves for sass consumers
  (verified against the packed tarball with both `@use "pkg:…"` + `--pkg-importer=node`
  and `@import` + `--load-path`). unpkg/jsdelivr fields point at existing files.
- `bin/release` runs tests before publishing and rebuilds docs/dist in one commit.

## A. High severity

### A1. LICENSE file is MIT, package.json says ISC
The legal text and the declared license disagree. npm, jsdelivr, and license
scanners report ISC while the actual grant is MIT. README does not disambiguate.

Repro:
```
$ head -1 LICENSE
The MIT License (MIT)
$ grep '"license"' package.json
  "license": "ISC",
```

### A2. Dart Sass 2.0/3.0 removal horizon: `@import`, `/` division, legacy `render()`
The entire tree (48 occurrences) uses `@import`, deprecated since dart-sass 1.80
and slated for removal in Dart Sass 3.0. Division with `/` (removed in 2.0) is
used in `src/smcss/parser/eval-size.sass:18` and `src/smcss/container/grid.sass:18`.
The test harness (`test/index.js:10`) uses the legacy `sass.render` callback API
(removed in 2.0). The pinned sass 1.72 masks the `@import` warnings; consumers on
sass ≥1.80 already get a wall of deprecation warnings, and the package stops
compiling entirely at Dart Sass 3.0.

*Addendum (found during Phase 3, sass 1.101):* two more deprecation classes the
1.72 toolchain could not surface — `global-builtin` (un-namespaced `str-slice`
and friends) and `if-function` (the sass `if()` function, which now clashes with
CSS `if()`). Both are part of the same Phase 4 migration; they are silenced in
the test suite (`test/index.js` `silenceDeprecations`) but still reported by
`bin/build`.

Repro:
```
$ grep -rn '@import' src/smcss index.sass demos/sm.sass | wc -l
48
$ ./node_modules/.bin/sass demos/sm.sass >/dev/null
DEPRECATION WARNING: Using / for division outside of calc() is deprecated ... (×5)
$ grep -n 'sass.render' test/index.js
```

### A3. The path that generates dist/sm.css has zero test execution
Tests exercise the `smcss()` mixin per-token, but the class-emission/export layer
that actually produces the published stylesheet — `parser/sm.sass`
(`sm`/`smx`/`smx-direct`/`smx-amp`), `parser/smcss-export.sass`, and
`helper/str-split.sass` — is never executed by any test. Nothing asserts that
`demos/sm.sass` compiles or that dist matches src; today's sync is discipline, not
a check. `src/smcss/experimental/` has zero coverage and isn't even imported.

Repro:
```
$ grep -c 'smx\|smcss-export\|str-split' test/index.js
0
$ grep -rn "demos/sm.sass" test/            # nothing
```

## B. Medium severity

### B1. No CI
No `.github/workflows` (no `.github` at all). Tests and src↔dist sync are
enforced only by `bin/release` on the maintainer's machine; a push with red tests
or drifted dist is invisible.

Repro: `ls -a /app | grep github` → nothing.

### B2. 13 npm-audit vulnerabilities in devDependencies (6 high, 6 moderate, 1 low)
picomatch (glob injection + ReDoS), serialize-javascript ≤7.0.4 via mocha,
postcss <8.5.10 (direct dev dep at 8.4.38), yaml 2.x via postcss-cli. Runtime
`dependencies` are empty, so **consumers are unaffected** — this is maintainer-
machine exposure only.

Repro: `npm audit 2>&1 | tail -5` → `13 vulnerabilities (1 low, 6 moderate, 6 high)`.

### B3. Abandoned/unnecessary test toolchain
`power-assert@1.6.1` (2018) and `intelli-espower-loader@1.1.0` (2017) are
unmaintained; `bluebird@3.7.2` is required at `test/index.js:1` solely to shadow
native `Promise` — the code only uses `new Promise(...)`, so the dependency is
dead weight. `mocha` is a major behind (10.4 → 11.x), `sass` 29 minors behind.

Repro: `npm outdated`; `head -1 test/index.js`; `npm view intelli-espower-loader time.modified`.

### B4. Published docs site has broken demo scripts (`../vue/dd.js`)
`demos/text/1.html` and `demos/components-range/1.html` load `../vue/dd.js`,
but `demos/vue/` contains only `input-height.js`, `input-width.js`, `mixins.js`.
Both pages are reachable from the published demos index (Preview buttons), so the
live site ships demos with a 404'd script.

Repro:
```
$ grep -rln 'dd.js' demos/
demos/components-range/1.html
demos/text/1.html
$ ls demos/vue/
input-height.js  input-width.js  mixins.js
```

### B5. README installs a stale version: unpkg pinned at @1.4.0
README.md:22-23 and demos/README.md:20-21 pin `@vbarbarosh/smcss@1.4.0` while the
package is at 1.5.1. Anyone copy-pasting the quick-start gets an old build.

Repro: `grep -rn '1.4.0' README.md demos/README.md`.

### B6. `bin/update-docs` destroys docs/ before checking its prerequisites
The script does `rm -rf docs` and only later invokes `php`. On a machine without
PHP (like this one), `set -e` aborts mid-way, leaving `docs/` half-built with no
`index.html` — and nothing documents the PHP requirement.

Repro:
```
$ php --version
bash: php: command not found
$ sed -n '/rm -rf docs/,/php src/p' bin/update-docs   # rm happens first
```

### B7. dist/reset.css is not a neutral reset — it styles all tables
The "/* vb */" section at the bottom of `reset/reset.css` forces
`table, td, th { border: 1px solid #aaa }` and `th { background: #ddd }` on every
table. Consumers pulling `dist/reset.css` expecting the Meyer reset get visible
debug borders on all tables. The base is Meyer reset v2.0 (2011); `main`,
`dialog`, `picture`, `template` are absent from the element list.

Repro: `tail -3 reset/reset.css`.

*Resolution (2026-07-14):* intentional, kept — the maintainer relies on bare
`<table>` markup being readable in quick snippets. The cryptic `/* vb */`
marker was replaced with a comment stating the intent, and the file header
notes the addition. Not a defect; plan item 14 is closed as won't-fix.

## C. Low severity

### C1. Invalid/no-op CSS locked in by the golden tests
- `pa` emits `padding: auto` — invalid, silently dropped by browsers (dist/sm.css:29146).
- `outline` emits `outline: 1px currentColor` — no style keyword, renders nothing;
  `border` uses `solid` (dist/sm.css:23266).
- IE-era artifacts still emitted: `*zoom: 1` star hack (×2), `-ms-flex-preferred-size`,
  `-moz-scrollbars-none`.
The test table asserts these exact strings (test/index.js:265, 250, 188-189), so
they can't regress silently — but they also can't be fixed without touching tests.

Repro: `grep -n 'padding: auto' dist/sm.css; grep -n 'outline: 1px currentColor' dist/sm.css; grep -c 'zoom: 1' dist/sm.css`.

### C2. `.idea/` untracked and not gitignored
Repro: `git status --short` → `?? .idea/`; `git check-ignore .idea` → exit 1.

### C3. `.browserslistrc` comment disagrees with its own config
Comment link says `Chrome >= 50`; the config line says `Chrome >= 40`. Config
unchanged since 2017 and still targets IE ≥ 10 / Safari ≥ 7, which is why
dist/sm.css carries 326 `-webkit-`/`-ms-` prefix lines.

Repro: `cat .browserslistrc`.

### C4. Silent misbuild hazard: browserslist resolved from postcss *output* path
`postcss -o <file outside the repo>` silently skips all IE/old-Safari prefixes
(config lookup follows the output file's directory when input is stdin),
producing a ~19 kB-smaller, prefix-free sm.css with no error. The npm script
writes inside the repo so today's build is correct, but nothing pins the config
(no `browserslist` key in package.json).

Repro:
```
$ printf '.h {\n  display: flex;\n}\n' | ./node_modules/.bin/postcss          # prefixed
$ printf '.h {\n  display: flex;\n}\n' | ./node_modules/.bin/postcss -o /tmp/y.css && cat /tmp/y.css  # NOT prefixed
```

### C5. Obsolete build docs in `src/smcss/index.sass:1-8`
Header documents the ruby-sass / sass-css-importer / global-autoprefixer
workflow; the real build is dart-sass + postcss via npm scripts.

Repro: `head -8 src/smcss/index.sass`.

### C6. Stale/orphaned template and doc sources
- `src/templates/app/package.json` uses `autoprefixer ^7.1.6` and dependency
  `"smcss": "github:vbarbarosh/smcss"` (pre-scope name); nothing references it.
- `src/templates/sm.sass` starts with a stub `@mixin smx` whose body is `// true`;
  not wired to any build.
- `src/docs/reference/toggle.php` documents a `toggle` group with no module in
  `src/smcss/` (only reference page without a module; every module has a page).
  *Resolution (2026-07-14):* kept — on inspection it is a deliberate,
  WIP-labelled concept page for the `.toggle`/`.if-on`/`.if-off` pattern (with
  a live demo in `demos/toggle/`), not a stale module reference. The
  `src/templates/` tree (2017 `app/` starter + unwired `sm.sass` stub) was
  deleted instead.

Repro: `grep -n 'autoprefixer\|smcss' src/templates/app/package.json; head -2 src/templates/sm.sass; comm -23 <(ls src/docs/reference/*.php | xargs -n1 basename | sed s/.php// | sort) <(ls src/smcss/*.sass | xargs -n1 basename | sed s/.sass// | sort)`.

### C7. Docs iframe link escapes the site root
`src/docs/reference/cursor.php:45` uses `../../../demos/cursor/index.html` from
`/reference/` — two levels above the site root. It works in browsers only because
they clamp `..` at the root; strict resolvers and local file:// preview break.

Repro: `grep -n 'demos/cursor' src/docs/reference/cursor.php`.

### C8. Tarball ships unreachable code
`src/smcss/experimental/` (cage, flow, table, transition + 3 css files) ships in
the npm tarball but its imports in `src/smcss/index.sass` are commented out.

Repro: `npm pack --dry-run 2>&1 | grep experimental; grep -n 'experimental' src/smcss/index.sass`.

### C9. package.json has no `engines` field
No declared Node range for contributors. Repro: `grep engines package.json` → nothing.

### C10. No minified build published
`dist/sm.css` is 338 kB / ~38k lines (brute-force enumerations `w0..w1200`,
`h0..h1200`, …). csso is already a devDependency; a `dist/sm.min.css` would be
nearly free. Gzip mitigates for CDN users, but direct downloads get the full file.

Repro: `ls -la dist/sm.css`.

### C11. `--bail` in the test script masks failure breadth
A broken shared code path reports as a single failure. Repro: `grep bail package.json`.

### C12. Repo weight from historical assets
`proto/` (768 kB, incl. vendored `jquery.min.js`/`vue.min.js`), `bugs/` (688 kB),

Repro: `du -sh proto bugs docs`.

## Improvement plan

Ordered by leverage; each phase is independently shippable.

### Phase 1 — Correctness & trust (hours)
1. **Fix the license mismatch** (A1): pick MIT (matches the LICENSE text),
   set `"license": "MIT"` in package.json, mention it in README. One-line patch.
2. **Fix README version pins** (B5): drop the `@1.4.0` pins in favor of
   `@latest`, or have `bin/release` sed the current version into README.
3. **Fix broken demo scripts** (B4): restore `demos/vue/dd.js` or remove the
   `<script>` tags from `text/1.html` / `components-range/1.html`; regenerate docs.
4. **Fix `outline` and `pa`** (C1): `outline: 1px solid currentColor`; make `pa`
   emit nothing or drop the token. Update the two golden-table entries in the
   same commit — the tests make this a safe, visible change.
5. Add `.idea/` to `.gitignore` (C2); align the `.browserslistrc` comment (C3).

### Phase 2 — Safety net (a day)
6. **Add CI** (B1): a GitHub Actions workflow running `npm ci && npm test` plus a
   dist-freshness check (`npm run build && git diff --exit-code dist demos/demo.css`).
   This turns the currently discipline-based src↔dist invariant into a machine check
   and closes A3's biggest practical risk with one job.
7. **Add one export-path test** (A3): compile `demos/sm.sass` in the test suite and
   assert it equals `dist/sm.css` (through the same postcss step), covering
   `sm`/`smx`/`smcss-export`/`str-split` in a single golden test.
8. **Guard `bin/update-docs`** (B6): `command -v php >/dev/null || { echo "php required"; exit 1; }`
   *before* `rm -rf docs`; document the PHP requirement in README.
9. Add `engines` (C9) and pin browserslist config into `package.json` (C4).

### Phase 3 — Dependency & toolchain refresh (a day)
10. **Modernize the test harness** (B3, part of A2): drop bluebird,
    intelli-espower-loader and power-assert (plain `assert.strictEqual` already
    carries the suite; mocha 11 prints usable diffs), switch `sass.render` to
    `sass.compileString`. Then `npm audit fix` / bump mocha, postcss, sass (B2).
    Consider removing `--bail` (C11) once failures are cheap to read.
11. Publish `dist/sm.min.css` via csso in the build script (C10).

### Phase 4 — Dart Sass migration (the big one, ~days)
12. **Migrate `@import` → `@use`/`@forward` and `/` → `math.div`** (A2). The
    sass-migrator tool automates most of it; the 468-test golden table plus the
    new dist-equality test (step 7) make this refactor verifiable. Do this after
    Phase 2/3 so the newest sass surfaces all remaining deprecations in CI.
13. Decide the fate of `src/smcss/experimental/` (C8): wire it in, or exclude it
    from `files`.

### Phase 5 — Docs & housekeeping (opportunistic)
14. Split `dist/reset.css` opinions (B7): move the table debug styles out of the
    reset (e.g. into an smcss `debug-tables` utility) or document them loudly.
15. Refresh or delete `src/templates/app/` and `src/templates/sm.sass` (C6);
    remove the orphan `toggle.php` or add the module it describes.
16. Fix the cursor iframe path (C7); rewrite `src/smcss/index.sass`'s header to
    describe the real build (C5); prune `proto/`'s vendored jquery/vue (C12).
17. Update `notes/06-tests-and-quality.md` with the corrections in C13.

### Phase 6 — parser speed (optional)

18. **Dispatch keyword tokens via a sass map instead of the linear if/else
    chain.** `parser/smcss.sass` resolves every token by walking ~440 branches
    top to bottom; keyword tokens (the majority) could live in a
    token → mixin map dispatched with `meta.apply`/`mixin-exists`, leaving the
    chain only for parameterized tokens (`w200`, `mt10n`, …). Baseline after
    the @use migration (sass 1.101, 2026-07-13): full-stylesheet compile
    (`demos/sm.sass`, 32k lines out) ~3s; 467 single-token test compiles ~9s.
    The win would show in `smx`-heavy builds, which evaluate the chain once
    per token per class. Verify with the golden table + dist-equality test;
    no output may change.

    *Measured 2026-07-14 and shelved:* branch cost is ~0.35µs (2000
    first-branch tokens: 508ms vs 2000 late-branch tokens: 791ms), so
    dispatch is only ~15-20% (~0.3-0.4s) of the 2s full build. The clean
    map dispatch needs `meta.apply` (sass 1.69+), which would break the
    documented sass 1.33 compatibility floor; a first-letter trie of nested
    `@if`s keeps the floor but reshuffles 900 lines for a marginal win.
    Revisit only if the token count grows an order of magnitude or the
    floor moves past 1.69 anyway.

    *Real-world profile (2026-07-14, measured on a large downstream app):*
    the production cost is not dispatch but **per-style-block library
    loading**: when many Vue/webpack components import the library in their
    style blocks, each block is a separate sass compilation and each pays
    ~70ms of smcss loading — tens of seconds per cold build at scale. Sass
    has no mixin/function memoization within a compilation, but dart-sass
    1.70+ shared compilers (`sass.initCompiler()`; sass-loader
    `api: 'modern-compiler'` + `sass-embedded`) cache parsed files ACROSS
    compilations (verified: ~35% faster even with the pure-JS sass package)
    — the fix belongs in the consumer's webpack config, not in smcss. A
    flattened single-file smcss distribution is the second-order option.
